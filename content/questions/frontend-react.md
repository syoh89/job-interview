### 질문1

컴포넌트를 어떻게 분리하고 책임을 나누나요?

### 답변1

재사용성과 변경 가능성을 기준으로 분리하고, 책임은 상태 변경 경계와 사용자 플로우를 따라 나눕니다. 재사용성이 높은 컴포넌트일수록 변화 가능성을 염두에 두고 더 작게, 더 독립적으로 설계합니다.

#### 이해한 과정1

재사용성은 곧 변경 가능성의 신호이므로, 구조를 작게 쪼개고 책임을 명확히 해야 합니다. 상태가 변하는 지점과 사용자 흐름의 단계가 컴포넌트 경계를 결정합니다.

#### 예시1

로그인 플로우 기준으로 책임을 나누면 다음 단계별로 컴포넌트 경계가 나뉩니다.

1. 로그인 페이지 접근 및 렌더링
2. ID/PASSWORD 입력
3. 제출(버튼 또는 엔터) 트리거
4. 입력값 처리 후 서버 검증 요청
5. 응답 결과에 따른 에러 표시 또는 리다이렉트

### 질문2

상태는 어떤 기준으로 상위/하위 컴포넌트에 둡니까?

### 답변2

상태를 공유하는 범위가 넓을수록 상위로 올리고, 한 컴포넌트에서만 쓰는 값은 하위에 둡니다. 리렌더 영향 범위를 최소화하려면 지역 상태는 가능한 좁게, 공유 상태는 필요한 공통 부모까지 끌어올리는 것이 기준입니다.

#### 이해한 과정2

상태는 화면을 구성하는 데이터이므로, 누가 그 데이터를 공유하는지와 변화가 어디까지 전파되어야 하는지를 먼저 따집니다. 공통으로 쓰는 값은 상위, 지역적으로만 쓰는 값은 하위로 두는 것이 가장 단순하고 안전합니다.

#### 참조

- Cursor 로 하여금 사용자에게 질문하게 한 후 사용자의 답변을 정리하여 문서화 하도록 생성.
